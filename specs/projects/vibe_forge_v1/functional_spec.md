# Functional Spec: Videowright

The project overview covers the why and the locked architecture. This document covers behavior, contracts, and edge cases — what someone unfamiliar with the project needs to know to understand what it does.

The product name is **Videowright** (resolves overview open question #7). "VideoForge" anywhere in the overview should be read as Videowright.

---

## 1. Scope

### v1 (this project)

- `packages/lib/` — the npm package: player runtime, segment helpers, timeline loader, built-in transitions, `script()` helper, public types.
- `skill/` — the Claude Code agent skill: SKILL.md + reference docs + templates.
- `examples/` — one minimal smoke-test video that exercises: multiple segments, internal beats, one custom transition, one Three.js *or* animated-SVG element. Not a polished reference.
- Documentation for screen-record export as a manual flow (no shipped command).

### Later phases (in implementation plan; may or may not ship)

- `videowright record` — automated screen-record export (Playwright drives full-screen player; ffmpeg captures the window).
- `videowright render` — deterministic CDP-driven frame export.
- Audio playback / TTS integration.
- Polished reference videos.
- View Transitions API / shared-element morphs (documented as v2 escape hatch only).

The export commands are explicitly the **last phase** of the implementation plan. Screen-record before CDP render. The plan must include them; whether they ship depends on time.

## 2. Consumer repo layout

The skill scaffolds this on first setup. **Flat by design** — there is no `shared/` folder. Anything in a top-level dir is reusable from anywhere without moving it.

```
some-team-videos/
├── package.json
├── videowright.config.ts             # config + setup marker (§6)
│
├── videos/                           # one folder per video
│   └── 2026-01-launch/
│       ├── timeline.ts               # composition + meta (§5)
│       ├── README.md                 # what this video is
│       ├── PLAN.md                   # agent's working memory
│       ├── voiceover/
│       │   ├── script.md             # generated by `script()` (§8)
│       │   └── audio/                # rendered VO (later phase)
│       └── exports/                  # MP4s, gitignored
│
├── segments/                         # all segments — any video can use any
│   ├── intro/
│   │   └── index.ts
│   └── launch-2026-problem/
│       └── index.ts
│
├── components/                       # web components used inside segments
├── transitions/                      # custom transition functions
├── styles/
│   ├── tokens.css
│   ├── tokens.ts
│   └── STYLE.md                      # design notes the agent reads
├── assets/                           # fonts, icons, footage, screenshots
└── .claude/
    └── CLAUDE.md                     # references the Videowright skill
```

**Rules:**

- Anything inside a top-level dir other than `videos/` is shared by definition. Reuse without moving.
- Per-video files that won't be reused (`timeline.ts`, `README.md`, `PLAN.md`, `voiceover/`, `exports/`) live inside `videos/X/`. There is no per-video `segments/`, `components/`, `transitions/`, `assets/`, or `styles/`.
- Naming inside top-level dirs is the user's responsibility (`segments/launch-2026-intro/` for one-off, `segments/intro/` for generic). Collisions are the user's problem.

## 3. Player

The runtime that loads a timeline and renders it. Single HTML page; vanilla Web Components + TypeScript strict.

### Modes

- **Interactive** — keyboard + mouse driven. Used for development and review.
- **Render** — scripted advance via each segment's `advances` array. Forward-only. Used by export tooling.

### Controls (interactive mode, v1)

| Action | Input |
|---|---|
| Advance (next internal beat or next segment) | `→`, `Space`, mouse click anywhere on the player |
| Go back | `←` |
| Restart from the first segment | `R` |
| Jump to nth segment in timeline | `1`–`9` |
| Jump to a specific segment id on load | URL query string `?to=<id>` |
| Toggle dev-mode HUD | `H` |

These keybindings are locked for v1. No customization mechanism.

### `next` semantics

- If the current segment's `next()` returned `true`, the segment handled the press internally; the player does not advance.
- Otherwise the player begins the transition to the next segment.
- At the last segment: no-op. HUD shows "end of timeline".

### `prev` semantics

- At the start of a segment: jump to the previous segment.
- At the first segment: no-op.
- **No animation reversal.** Going back rewinds to a stable starting state via remount. Animations are never played in reverse. The player is a dev tool, not a render tool.
- Custom `prev()` on a segment may consume the press (return `true`) to handle internal back-navigation; otherwise the player handles it as above.

### Render mode

- Forward only. `prev`, `pause`, `restart`, `jump-to`, mouse click, and HUD-toggle are not honored.
- Each segment's `advances` array drives advancement; the driver fires `triggerNext()` at the scheduled times. `ctx.waitForNext()` resolves on each scheduled advance.
- Runtime coherence checks detect mismatches: if a segment transitions before all advances fired, or parks on `waitForNext` after all advances fired, the driver errors with the segment id and an actionable message.

### Dev-mode HUD

Toggleable overlay (`H`) showing:

- Current segment id
- Time within segment (controlled clock)
- Total elapsed time (controlled clock)
- Voiceover text for the current segment
- Mode (interactive / render)
- Keyboard shortcut reference
- "End of timeline" indicator when playback has ended

On by default in interactive mode; off by default in render mode.

### Hot reload

`videowright dev` watches the timeline file, segment files, transitions, and config; on change it does a full reload of the player.

The player continuously reflects current position in the URL (segment id + internal beat index) via the URL hash, e.g. `#/intro/2`. On load the player restores from the hash if present. After hot reload the user lands back on the same segment + beat.

The same URL state is what the existing `?to=<id>` query string addresses (§3 controls); the two are unified — `?to=<id>` is shorthand for setting the hash on load.

## 4. Segment contract

The interface is locked in the project overview. This section adds behavioral details.

### Lifecycle

1. `mount(el, ctx)` — called when the player gives the segment its element. Optional. Use to render initial DOM.
2. `play(ctx)` — called once `mount()` resolves. Returns when the transition out can begin.
3. `unmount()` — called when the segment is removed (transition complete, or `prev`/`jump-to` navigation, or hot reload). Cleanup point. Optional.

### `play()` rules

- **Must be idempotent across remounts.** A segment may be unmounted and remounted within a session via `prev`, jump-to, or hot reload.
- **Must use `ctx.waitForNext()` and `ctx.hold(ms)` for timing.** Not `setTimeout`, not `new Promise(r => setTimeout(r, x))`.
- **Must respect `ctx.signal`** for any work that outlives `play()` resolution. Pass it to `fetch`, abort GSAP timelines on `signal.aborted`, etc.

### Internal beats

Calling `ctx.waitForNext()` N times in `play()` creates N internal beats. Each call corresponds to one user advance press (interactive) or one scheduled advance from the segment's `advances` array (render/record).

Authors may override `next()` / `prev()` on the segment for custom logic:

- Returning `true` consumes the press; player does not advance.
- Returning `false` or `undefined` lets the player advance.

### `defineSegment` helper

The library exports `defineSegment(spec)` in v1. It wires `ctx.waitForNext()` to internal-beat tracking and the auto-generated default `next()`, sets up the abort signal, and returns a typed segment. Authors who prefer the bare interface still get strict typing.

```ts
export default defineSegment({
  id: 'intro',
  voiceover: '...',
  async play(ctx) { /* ... */ },
});
```

## 5. Timeline file contract

A timeline is a TypeScript module exporting a default object:

```ts
export default {
  meta: {
    title: string;                                            // required
    aspectRatio?: '16:9' | '9:16' | '1:1' | string;          // default from config
    resolution?: [number, number];                            // default from config
    fps?: number;                                             // default from config
  },
  segments: Array<{
    id: string;
    transition?: string | { type: string; [k: string]: unknown };
  }>,
};
```

### Segment id resolution

`{ id: 'intro' }` resolves to `segments/intro/index.ts` (relative to repo root). **Single lookup. No fallback chain.** If the file does not exist, the player errors at load with the missing id and the timeline path.

### Reordering

Reordering = moving array entries. Segments configure themselves; the timeline file does not pass props to segments.

### Meta inheritance

Missing fields on `meta` fall back to `videowright.config.ts` defaults. `meta.title` is the only required field.

## 6. Configuration: `videowright.config.ts`

Single config file at the consumer repo root. Serves as both **setup marker** (its presence signals the project is initialized) and **repo-wide configuration**. There is no `.video_forge.yml` or other yaml config.

```ts
import { defineConfig } from 'videowright';

export default defineConfig({
  projectStructure: 'v1',           // setup marker; bumped when layout changes
  defaults: {
    resolution: [1920, 1080],
    fps: 60,
    aspectRatio: '16:9',
  },
  transitions: {
    'morph-logo': () => import('./transitions/morph-logo'),
  },
});
```

Out-of-the-box defaults: `1920x1080`, 60fps, `16:9`.

## 7. Transitions

Transitions happen *between* segments, not *during*. In-segment animation goes inside the segment.

### Built-ins (v1)

- `fade` — opacity crossfade.
- `slideLeft`, `slideRight`, `slideUp`, `slideDown` — directional slide.
- `cut` — instantaneous swap (no animation).

Each ~20 lines using the Web Animations API.

### Custom transitions

User code in `transitions/<name>.ts`, registered in `videowright.config.ts`, referenced by name in timeline:

```ts
{ id: 'product-demo', transition: 'morph-logo' }
```

Or as a configured object:

```ts
{ id: 'product-demo', transition: { type: 'morph-logo', shared: 'logo' } }
```

### View Transitions API

Documented as a v2 escape hatch for shared-element morphs. **Not implemented in v1.** The two mechanisms (player-orchestrated transitions vs. browser-native View Transitions) are different problems and are not unified.

## 8. Voiceover

- Each segment optionally declares a `voiceover: string` field.
- The library exports a `script(timeline)` helper that walks segments in order and concatenates VO into a single Markdown document with segment-id headers + VO body, suitable for review or handoff to TTS.
- The HUD displays the current segment's VO text in interactive mode.
- v1 ships **no audio playback**. The script document is the only VO output.

## 9. CLI surface (v1)

Two commands, `videowright`:

| Command | Behavior |
|---|---|
| `videowright dev [timeline-path]` | Opens the player against the timeline. Defaults to discovering the most recent video by mtime. Live-reloads on file change. |
| `videowright script [timeline-path]` | Emits the concatenated VO script to stdout. With `--write`, writes to `videos/X/voiceover/script.md`. |

**Not in v1:**

- `videowright new`, `new-video`, `new-segment` — the skill scaffolds via `mkdir` / `Write` directly. CLI surface stays small.
- `videowright record` and `videowright render` — later phases.

CLI errors with a clear message (and the location it expected) if `videowright.config.ts` is missing.

## 10. Skill behaviors (v1)

### 10.1 Setup flow

When a user request requires the consumer repo to exist (anything that touches videos or segments) and `videowright.config.ts` is missing, the skill enters an **interactive setup flow**. The bar: great onboarding, but agent-native — drive the conversation efficiently, ask one focused round, scaffold in one batch.

The flow does roughly this. Exact prompts are an implementation concern; the shape is what's locked.

1. **Confirm intent.** Ask before scaffolding anything. Never auto-scaffold.
2. **Ask for the first video's name.** Default `demo_video` offered if the user has nothing in mind. Date prefix suggested but not enforced.
3. **Ask about specific dependencies the user wants set up.** Open-ended, default none. Anchor the user with example answers rather than a checklist — e.g. "Raw HTML", "Three.js for animations", "Lottie for logos", "React and shadcn to emulate our app UI". The agent installs whatever the user asks for using its general knowledge of those tools; there are no pre-built starter templates to maintain. The exact prompt is in the skill (see component skill.md), not locked here.
4. **Scaffold the layout.** Create the §2 directory structure plus `videowright.config.ts`.
5. **Create a 2-segment hello-world video** at `videos/<chosen-name>/`. Includes `timeline.ts`, `README.md`, `PLAN.md`, two example segments (in top-level `segments/`, wired into the timeline), and a stub `voiceover/script.md`. Templates ship with the skill. The hello-world must run immediately — `videowright dev` shows working content.
6. **Confirm.** Report what was created and point at `videowright dev`.

This flow's prompt copy and exact ordering are not locked; treat them as ideas to iterate on. The quality bar is locked: a fresh user with `npm install videowright` reaches a working first video in under 2 minutes via the skill.

### 10.2 Other skill behaviors

The skill also teaches the agent to:

1. **Scaffold a video** beyond setup. Create `videos/<name>/` with `timeline.ts`, `README.md`, `PLAN.md`, `voiceover/`.
2. **Scaffold a segment.** Create `segments/<id>/index.ts` using the canonical `defineSegment(...)` style with `waitForNext` / `hold`. No `duration` field, ever.
3. **Scaffold a component / transition / styles entry.** As needed, in their top-level dirs.
4. **Maintain `PLAN.md`.** Read before iterating on a video; append after meaningful changes (script revisions, design choices, user feedback).
5. **VO-first authoring.** Take a VO script, scaffold segments to match.
6. **Style matching.** When asked to make a video matching a past one's look, read `styles/STYLE.md`, the past video's `timeline.ts`, its segments, and its `PLAN.md`, then produce the new content.
7. **Document the `setTimeout` footgun.** Tell the author to prefer rAF / CSS / WAAPI / GSAP for any animation, since `setTimeout`/`setInterval` won't honor render-mode clock control.
8. **Document "any web tech welcome inside segments."** Three.js, animated SVG, Lottie, shadcn/React, GSAP — explicitly encouraged. The segment owns its DOM; isolation via shadow root is a per-segment choice, not a lib default.

The skill's internal folder structure (SKILL.md vs reference docs vs templates) is an architecture concern.

## 11. Library public surface

Functions and types exported from the `videowright` npm package:

| Export | Purpose |
|---|---|
| `defineSegment(spec)` | Author a segment with internal-beat tracking and signal plumbing. |
| `defineConfig(config)` | Author a typed `videowright.config.ts`. |
| `script(timeline)` | Walk a timeline and produce the concatenated VO Markdown document. |
| `Player` | The player runtime. Mounts to a host element; loads a timeline. |
| Built-in transitions | `fade`, `slideLeft`, `slideRight`, `slideUp`, `slideDown`, `cut`. |
| Types | `Segment`, `PlayerContext`, `Timeline`, `TimelineMeta`, `Transition`, `Config`. |

Published as ESM. The minimum-viable consumer path (drop an HTML file, open it directly) must work without a build step for trivial cases.

## 12. Edge cases & error handling

| Situation | Behavior |
|---|---|
| Timeline references missing segment id | Player errors at load with the id and timeline path. |
| Segment's `mount()` or `play()` throws | Player surfaces the error in HUD and stops; does not advance. Stack trace in console. |
| `advances` count ≠ actual presses needed | Render/record drivers detect the mismatch at runtime via coherence checks and error with the segment id and an actionable message. Interactive mode ignores `advances`. |
| User presses next at end of timeline | No-op. HUD shows "end of timeline". |
| User presses prev at first segment | No-op. |
| `setTimeout` / `setInterval` inside a segment | Works in interactive. Will desync from controlled clock in render mode (later phase). Documented footgun. |
| Asset fails to load (image, font, video) | Segment owns its own loading. Library does not impose a loader. Author handles. |
| Config file missing | CLI errors with "no `videowright.config.ts` found; the skill can scaffold one". |
| Config `projectStructure` doesn't match library version | CLI errors with a migration hint. |
| Two segments share an id | Filesystem prevents this — they live at the same path. |

## 13. Constraints

- **Browsers.** v1 targets recent Chrome / Edge / Safari / Firefox for the interactive player. Render mode (later phase) is Chrome / headless Chromium only (CDP).
- **TypeScript.** Strict mode in lib, skill scaffolds, and example.
- **Tests.** Unit tests as we go. At minimum, an integration test that loads a sample timeline, walks it forward, and verifies segment lifecycle calls.
- **Lint / format.** Configured in repo, run in CI.
- **CI.** A dedicated implementation phase wires up CI to run tests + lint + type-check on every push.
- **Build step.** Lib is published as ESM. Consumer repos may use any toolchain. The simplest case (no build step) must work for trivial videos.
- **License.** This project ships under the **MIT license**. All runtime dependencies must be MIT-compatible (no BUSL, no copyleft). This is the killer reason for not using Remotion.
